# Database Guide

This guide documents the database as implemented in **v0**.

## Role of PostgreSQL

PostgreSQL stores application data for the local-first MVP.

The backend is the only layer that should connect to PostgreSQL. The frontend must use FastAPI routes instead of direct database access.

## SQLAlchemy Decision

The project uses SQLAlchemy 2.x sync ORM style for v0.

Reasons:

- simpler for a first full-stack project
- enough for the local-first MVP
- works cleanly with FastAPI service/repository structure
- can be changed later if async database access becomes necessary

## psycopg 3 Decision

The backend uses `psycopg` 3 as the PostgreSQL driver.

The expected SQLAlchemy URL format is:

```txt
postgresql+psycopg://USER:PASSWORD@HOST:PORT/DB_NAME
```

## Alembic Migration Workflow

Alembic manages database schema changes.

A migration is a versioned change to the database. Migrations let the database schema evolve predictably as models change.

Common commands:

```bash
cd backend
uv run alembic revision --autogenerate -m "describe change"
uv run alembic upgrade head
uv run alembic downgrade -1
```

## Migrations in v0

There are **five migrations**: the initial `faff5d463169_create_initial_tables.py` (never hand-edit it); Phase 12's `b3074aa5c807_demo_sessions_and_llm_calls.py`, which added an indexed `session_id` column to `holdings` / `watchlist_items` / `reports` / `chat_sessions` (with `server_default='local'`, backfilling all pre-demo rows into the local bucket) and created the `llm_calls` observability table; Phase 13's `86ba8cdd8e97_llm_calls_cached_tokens.py` (nullable `cached_tokens` for provider cache-hit counts); and Phase 14's `c953934595fd_document_chunks_table_for_rag.py` (idempotent `CREATE EXTENSION vector` + the `document_chunks` table with a `vector(1536)` column — dev Postgres is the `pgvector/pgvector:pg16` image) and `80ad439a5932_fts_gin_index_on_document_chunks.py` (GIN expression index matching the lexical-pool query verbatim). A new autogenerated Alembic migration is only required when a model's columns actually change.

## Index Guidance

Add indexes where they support obvious future access patterns:

- ticker lookups
- unique settings keys
- unique cache keys
- chat message session lookups

Avoid over-indexing before real query patterns exist.

## Usage Notes (v0)

- `market_data_cache` backs the shared cache-aside layer (`core/cache.py`) used by market data (~2 min quote TTL), news (~15 min), and financials (~1 day).
- `reports`, `chat_sessions`, `chat_messages` are written by the `ai/` and `chat/` modules.
- `holdings`, `watchlist_items`, `reports`, `chat_sessions` carry a `session_id` demo bucket (see `core/demo_session.py`); locally everything is `"local"`.
- `llm_calls` records one row per LLM call (prompt/completion/cached tokens, latency; the agent path fills `route='agent'` + the per-call `steps` index; embedding batches log as `kind='embed'` with the embedding model name) — written by `ai/llm_client` and `rag/embeddings_client`, aggregated by `GET /api/stats`.
- `document_chunks` (Phase 14) is the shared read-only RAG corpus — not session-scoped, like `market_data_cache`. One row per chunk: source URL/title/date, ticker, `chunk_index`, verbatim content, `vector(1536)` embedding. Chunk identity `(source_url, chunk_index)` is deterministic; re-ingest replaces a URL's chunks wholesale.
- `settings` is a key-value store; v0 uses it for the LLM master switch (`llm_enabled_until`). `stock_notes` has no API yet (reserved).
- In Docker Compose, `alembic upgrade head` runs automatically before the API starts.
